# CS 이슈/대응 기반 카테고리 자동 분류 웹 MVP

CS 전화문의 또는 상담 기록에서 추출된 이슈와 대응 텍스트를 입력하면, 규칙 기반 Agent가 가장 적절한 카테고리를 추천하는 Streamlit MVP입니다.

## 실행 방법

```bash
pip install -r requirements.txt
streamlit run app.py
```

명령어는 Windows의 PowerShell, 명령 프롬프트, VS Code 터미널, 또는 Codex 터미널처럼 명령어를 입력할 수 있는 터미널에 입력합니다.

현재 `localhost` 앱은 실행한 컴퓨터 안에서 열리는 로컬 사이트입니다. 오늘 노트북 전원을 끄면 앱도 멈춥니다. 내일 같은 노트북에서 다시 보려면 프로젝트 폴더로 이동한 뒤 다시 실행합니다.

```bash
cd C:\Users\김민서\Documents\Codex\2026-07-02\new-chat-2\cs-category-agent
streamlit run app.py
```

다른 노트북에서 보려면 GitHub에서 코드를 받은 뒤 그 노트북에도 Python 의존성을 설치하고 같은 명령으로 실행해야 합니다.

## 시험판 목표

이 앱은 CS 담당자 3명이 녹음본 또는 전사문을 기반으로 이슈/대응을 정리하고, 카테고리를 확인한 뒤, 책임 주체별 메일 초안을 만들 수 있는 시험판입니다.

현재 단계에서는 팀원 시연 안정성이 가장 중요하므로 실제 STT보다 샘플 전사문과 직접 입력 전사문을 우선합니다. 자동 처리 결과는 확정값이 아니라 CS 담당자가 검토할 초안입니다.

## CSV 누적 데이터와 알고리즘 개선

분류를 실행하면 결과가 `data/feedback/classification_results.csv`에 저장됩니다. 이 CSV는 실제 상담 데이터가 될 수 있으므로 GitHub에 커밋하지 않도록 `.gitignore`에 포함되어 있습니다.

저장되는 주요 컬럼은 다음과 같습니다.

- `created_at`: 분류 실행 시각
- `issue`: 입력된 이슈
- `response`: 입력된 대응
- `recommended_category`: Agent가 추천한 카테고리
- `final_category`: 상담사가 확정하거나 수정한 최종 카테고리
- `confidence`: 분류 신뢰도
- `needs_human_review`: 사람 검토 필요 여부
- `matched_keywords`: 분류에 사용된 키워드

이 데이터는 세 단계로 알고리즘 개선에 활용할 수 있습니다.

1. `recommended_category`와 `final_category`가 다른 케이스를 모아 자주 틀리는 유형을 확인합니다.
2. 틀린 케이스에서 반복되는 표현을 `data/categories.json`의 키워드에 추가하거나 가중치 규칙을 조정합니다.
3. 충분한 데이터가 쌓이면 `issue + response`를 입력값으로, `final_category`를 정답 라벨로 사용해 가벼운 머신러닝 분류기나 LLM 분류기로 교체할 수 있습니다.

## 무료 STT 방향

OpenAI STT API는 비용 문제로 기본 경로에서 제외합니다. 시험판은 샘플 전사문과 직접 입력 전사문으로 전체 흐름을 먼저 검증합니다.

향후 녹음본 STT를 붙일 때는 무료 로컬 모델을 우선 검토합니다.

- 1순위 후보: `faster-whisper`의 `tiny` 또는 `base` 모델
- 보조 후보: `whisper.cpp`
- 저사양 후보: `Vosk`

Windows 노트북 CPU 환경에서는 긴 통화 녹음본 처리가 느릴 수 있으므로, 초기에는 짧은 녹음본과 샘플 전사문으로 검증합니다.

## 향후 자동 알림 확장

카테고리 분류 결과를 바탕으로 관련 팀에 자동 메일을 보내는 기능도 확장 후보입니다.

예시는 다음과 같습니다.

- `[EB팀] 판독 관련 요청`: EB팀 담당 메일로 이슈/대응/추천 근거 전송
- `[앱] 관련 이슈`: 앱 개발 또는 운영 담당 팀에 전송
- `[웹] 관련 이슈`: 웹 담당 팀에 전송
- `[검토 필요]`: CS 리드 또는 운영 담당자에게 검토 요청

운영 적용 전에는 메일 자동 발송 대신 초안 생성과 담당자 확인 단계를 먼저 두는 것이 안전합니다.

## 분류 방식

`src/classify.py`의 `classify_issue_response()`가 이슈와 대응 텍스트를 분석합니다.

- 이슈 텍스트 키워드 가중치: 2.0
- 대응 텍스트 키워드 가중치: 1.0
- 점수가 낮으면 `[검토 필요]`
- 1위와 2위 점수가 비슷하거나 검토 신호가 있으면 사람 검토 권장

LLM 분류기로 확장할 때도 같은 함수 인터페이스를 유지하면 앱 변경을 줄일 수 있습니다.
